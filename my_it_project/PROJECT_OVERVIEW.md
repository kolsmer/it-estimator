# IT Estimator: обзор проекта

## 1. Что это за проект

`IT Estimator` это система для оценки IT-проектов.
Её задача: принять описание проекта или входной файл, разобрать требования, посчитать примерную оценку проекта и показать результат в удобном интерфейсе.

Проект состоит из двух частей:

- `it-estimator-backend` — backend на `FastAPI`
- `it-estimator-frontend` — frontend на `Next.js`

Система позволяет:
- анализировать текстовое описание проекта;
- анализировать загруженный файл;
- рассчитывать проектную оценку;
- сохранять результаты;
- просматривать проекты через UI;
- выгружать данные;
- архивировать, восстанавливать и удалять проекты;
- работать со ставками ролей.

## 2. Основная идея работы

Логика проекта в общем виде такая:

1. Пользователь вводит текст проекта или загружает файл.
2. Backend извлекает из этого входа структуру проекта.
3. Модуль оценки рассчитывает:
   - роли,
   - часы,
   - стоимость,
   - длительность,
   - этапы проекта.
4. Дополнительно формируется LLM-анализ с пояснениями.
5. Результат сохраняется в базу.
6. Frontend показывает список проектов, детали проекта, роли, архив и действия над проектами.

## 3. Из чего состоит система

## Backend

Backend реализован на `FastAPI`.

Основные задачи backend:
- принимать запросы от frontend;
- выполнять анализ входных данных;
- запускать оценку проекта;
- сохранять результаты;
- отдавать список проектов и детали;
- обрабатывать экспорт;
- управлять статусами проектов;
- обновлять настройки ставок.

Ключевые файлы backend:
- `app/api.py` — основной API-слой
- `app/internal/repository/project_storage.py` — работа с PostgreSQL
- `app/parser_agent.py` — извлечение структуры проекта из входного текста
- `app/estimator_moduleF.py` — расчёт оценки
- `app/aistart_with_llm.py` — работа с LLM
- `app/project_estimator.json` — конфигурация ролей, уровней и ставок
- `app/internal/repository/migrations/` — миграции PostgreSQL

## Frontend

Frontend реализован на `Next.js 14` с `App Router`, `TypeScript`, `Tailwind CSS`.

Основные задачи frontend:
- показывать dashboard;
- выводить список проектов;
- показывать карточку проекта;
- выводить роли и ставки;
- показывать архив;
- вызывать backend API для действий над проектами.

Ключевые каталоги frontend:
- `app/` — страницы приложения
- `components/` — UI и layout-компоненты
- `lib/api.ts` — клиент для вызова backend API
- `lib/` — утилиты и вспомогательная логика

## 4. Как работает backend

## 4.1. Входные сценарии

Backend поддерживает два основных сценария анализа:

### 1. Анализ текста
Эндпоинт принимает текстовое описание проекта и запускает расчёт.

### 2. Анализ файла
Эндпоинт принимает файл или содержимое файла, извлекает из него текст и затем запускает ту же логику анализа.

Поддерживаются:
- текст
- `json`
- `csv`
- `docx`
- `pdf`

## 4.2. Общий pipeline обработки

При анализе backend делает примерно следующее:

1. Получает входные данные.
2. Извлекает текст, если вход пришёл в виде файла.
3. Передаёт текст в модуль `parser_agent`.
4. `parser_agent` формирует структуру проекта.
5. Далее вызывается модуль оценки `EstimationFacade`.
6. На основе структуры проекта формируется расчёт.
7. Создаётся prompt для LLM.
8. Получается текстовое пояснение от модели.
9. Результат сохраняется в базу.
10. Готовый ответ возвращается клиенту.

## 4.3. Что хранится в проекте

В базе сохраняются:
- название проекта;
- тип проекта;
- стоимость;
- длительность;
- размер команды;
- дата генерации;
- исходные данные проекта;
- рассчитанная оценка;
- текст LLM-анализа;
- статус проекта;
- дата архивирования.

## 4.4. Основные API-методы backend

Примерная структура API:

- `GET /projects` — получить список проектов
- `GET /projects/archived` — получить список архивных проектов
- `GET /projects/{id}` — получить детали проекта
- `POST /projects/{id}/archive` — архивировать проект
- `POST /projects/{id}/restore` — восстановить проект
- `DELETE /projects/{id}` — удалить проект
- `GET /projects/{id}/export` — экспортировать проект
- `GET /roles` — получить агрегированные роли
- `GET /settings/rates` — получить ставки
- `PUT /settings/rates` — обновить ставку
- `POST /analyze` — анализ текстового описания
- `POST /analyze-file` — анализ файла

## 5. Как работает frontend

Frontend представляет собой интерфейс для просмотра и управления расчётами.

Основные разделы:
- `Dashboard`
- `Projects`
- `Project Details`
- `Roles`
- `Archive`

## Dashboard
Показывает общую сводку:
- количество проектов;
- количество активных и архивных;
- суммарные показатели;
- быстрые ссылки.

## Projects
Показывает таблицу проектов и действия:
- открыть проект;
- архивировать;
- экспортировать.

## Project Details
Показывает детали одного проекта:
- общие метрики;
- описание;
- LLM-анализ;
- роли;
- этапы;
- экспорт.

## Roles
Показывает агрегированную информацию по ролям и ставкам.

## Archive
Показывает архивные проекты и действия:
- восстановить;
- удалить.

## 8. Текущие технические ограничения

Проект ещё не доведён до полностью зрелого состояния.

Основные ограничения:
- backend содержит слишком много логики в крупных файлах;
- не хватает тестов;
- документация неполная;
- часть текстов и README имеет проблемы с кодировкой;
- не до конца оформлена эксплуатация и деплой;
- миграции базы данных пока минимальные и требуют расширения под production-сценарии;
- часть логики смешана между инфраструктурным и бизнес-слоем;
- нужна более явная архитектурная декомпозиция.

## 9. Что нужно понимать стажёру в первую очередь

Если смотреть на проект как на систему, важно понять следующее:

1. Это не просто UI, а система расчёта оценки проектов.
2. Backend — основной центр логики.
3. Frontend в основном отображает данные и управляет действиями.
4. Ключевая ценность проекта — pipeline:
   входные данные -> парсинг -> расчёт -> LLM-анализ -> сохранение -> отображение.
5. Главный технический риск сейчас не в отсутствии кода, а в незавершённости архитектуры, документации и проверки сценариев.

## 10. Что нужно сделать при изучении проекта

При первом входе в проект стажёру нужно:

1. Понять структуру backend и frontend.
2. Поднять backend локально.
3. Поднять frontend локально.
4. Проверить их совместную работу.
5. Пройти основные сценарии вручную.
6. Посмотреть ключевые файлы:
   - backend `api.py`
   - backend `db_repository.py`
   - backend `parser_agent.py`
   - backend `estimator_moduleF.py`
   - frontend `lib/api.ts`
   - frontend страницы в `app/`
7. После этого составить техническое заключение:
   - что работает;
   - что не работает;
   - что нужно исправить в первую очередь.

## 11. Ожидаемый результат от изучения

После ознакомления с проектом стажёр должен уметь ответить на вопросы:

- что делает система;
- как проходит путь данных внутри проекта;

- какие модули наиболее важны;

`IT Estimator` — это система для оценки IT-проектов с backend на `FastAPI` и frontend на `Next.js`.

Сейчас проект уже реализует основную бизнес-идею:
- принимает описание проекта,
- считает оценку,
- сохраняет результат,
- показывает его в интерфейсе.

При этом проект ещё требует технической доработки:

- эксплуатационной,
- тестовой,
- документальной.

Главная задача при дальнейшем развитии — превратить текущий рабочий каркас в стабильный, понятный и поддерживаемый продукт.
